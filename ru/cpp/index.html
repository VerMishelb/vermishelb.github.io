Настройка рабочей среды

IDE / редактор кода

Каждый использует то, что ему удобнее. Что под Windows, что под Linux можно использовать комплект из Visual Studio Code в качестве перередактора недо-IDE да CMake в качестве системы сборки (точнее, системы сборки для системы сборки, которая будет зависеть от операционной системы). VSC прекрасно работает на обеих системах, терпимо работает на слабых устройствах (я написал свой диплом на Fujitsu Lifebook UH552 и выжил).

Тем не менее, если вас понесёт в обычную Visual Studio, то следует учесть, что она не знает, что такое UTF-8. Чтобы VS нормально сохраняла ваши исходники, нужно в корне папки с проектом положить небольшой текстовый файлик с именем .editorconfig (без расширения .txt) со следующим содержимым: 

«.editorconfig»
root = true

[*]

charset = utf-8

Данный файл попросит VS сохранять и создавать вообще все файлы в UTF-8. Вряд ли через 10 лет появится что-то, что способно заменить UTF-8 в универсальности.

«А зачем это, собственно, нужно?», спросите вы. А у вас в коде будет Кириллица? Другие языки? Рано или поздно будет и то, и другое. Взять, хотя бы, вывод путей к файлам в программе. И первое, что сделает Windows — превратит вашу Кириллицу в Windows-1251. Код не перенесёт таких издёвок. Как только вы поймёте, что у вас проблемы с выводом, начнётся увлекательное путешествие в недры интернета, где вопросы на stackoverflow от 2009-го остаются без конкретного ответа, потому что нормального способа это решить так и не придумали.

К счастью, бо́льшая часть этого решается C++23.

Компилятор

Здесь простор для творчества!
На Windows можно поставить Visual Studio Build Tools, либо саму Visual Studio, т.к. Build Tools — её компонент. Компилятор Microsoft сокращают до MSVC (Microsoft Visual C++).
Кроме него есть GCC (GNU Compiler Collection), который только для Linux. Для винды есть порт MinGW, он же в MSYS2, он же в Cygwin. Если не MSVC, то ваш выбор скорее всего падёт на MSYS2. Установка, правда, страшная.
Ещё есть Clang. Он есть. Больше я о нём ничего сказать не могу, потому что не приходилось использовать.

Установка MSVC — прощёлкивание «Далее» несколько раз. Установка GCC на Linux — apt install gcc, или pacman -Syu gcc, или ещё как-нибудь похоже, исходя из вашего пакетного менеджера. Установка пакета gcc должна установить g++ — сам компилятор для C++.

Сборка и её автоматизация

Компилятор превращает рукописи в исполняемый код, но его придётся вызывать вручную. Допустим, у вас всего один файл кода «main.cpp». Тогда для сборки через GCC вам всего лишь нужно написать в консоль: 

$ g++ main.cpp -o main.elf
Теперь вы можете вызвать ./main.elf из терминала и получить вывод вашей программы. Здорово? Здорово.
.elf здесь не просто потому что я так решил, а потому что так называется формат исполняемых файлов в Unix. Это легко проверить, если вызвать на него $ file. Я был сильно удивлён, когда собрал программу по привычке с расширением (моя основная система — Windows 10), а позже получил вопрос от пользователя Linux про то, что значит такое расширение.

А теперь представьте, что у вас десяток таких файлов с кодом. Может быть, вы делаете игру? Тогда вам понадобятся сторонние библиотеки, а их тоже нужно собирать! Команда сборки превращается в стену текста: 

$ g++ -o "epic game.elf" main.cpp Entity.cpp Collision.cpp Window.cpp <и так далее>
Как быть? Нужно автоматизировать это дело. В Visual Studio за нас это делает сама ИСР (видели когда-нибудь аббревиатуру IDE на русском?). Без IDE нужно страдать. Самое просто решение — скрипты. Можно автоматизировать сборку с помощью bat или sh скриптов, в зависимости от ОС, и многие старые проекты спокойно с этим работают. Такое решение крайне ограничено в функционале, а ошибка в скрипте может привести к концу света. Тогда люди решили, что конец света им не нужен, и придумали Makefile. Теперь у них две проблемы. 

«/project/Makefile»
all:
    g++ main.cpp Entity.cpp Collision.cpp ... -o "epic game.elf"
clean:
    rm -rf *.o "epic game.elf"
Структура данного файла предельно проста, и в этом заключается главная сложность. В первой строчке указывается «цель» (all) и её зависимости после двоеточия, т.е. цели, которые должны быть выполнены до сборки выбранной цели. Следом, с табуляцией, идёт сама команда. Обратите внимание на пробел в названии выходного файла «epic game.elf». Огромное (ОГРОМНОЕ!!!) количество программ не могут справиться с пробелами в путях. Иногда кавычки спасают, как здесь, но полагаться на это не стоит. Кроме того, зачастую эти же программы не могут переварить кириллицу в путях.

При создании проектов выбирайте путь без кириллицы и пробелов!

С этого момента наша игра больше не «epic». Оставим её как game.elf для простоты. Структура проекта сейчас выглядит как-то так:
📁 project
 ├📄 main.cpp
 ├📄 Entity.cpp
 ├📄 Collision.cpp
 ├...
 └📜 Makefile
Теперь вместо стены текста проект собирается простым $ make all в папке с проектом. Мейкфайлы могут быть более навороченными: 
«/project/Makefile»
CC=g++
CFLAGS=-O3 -Wall -c
OUTPUT=game.elf

all: $(OUTPUT)

$(OUTPUT): main.o Entity.o Collision.o
    $(CC) main.o Entity.o Collision.o -o $(OUTPUT)

main.o: mainc.cpp
    $(CC) $(CFLAGS) main.cpp

Entity.o: Entity.cpp
    $(CC) $(CFLAGS) Entity.cpp

Collision.o: Collision.cpp
    $(CC) $(CFLAGS) Collision.cpp

clean:
    rm -rf *.o $(OUTPUT)
Это уже выглядит страшнее (я даже не уверен, что оно будет работать, т.к. писал их раза два в жизни).

— А зачем цель «clean»?

Иногда, во время сборки, часть объектных файлов (.o), в которые собираются наши исходники, может оказаться устаревшей или несовместимой с другими объектными файлами, а make по какой-то причине этого не заметил. Данная проблема встречается и в IDE, абсолютного спасения от неё нет. В случаях, когда ваш проект ведёт себя крайне странно, попробуйте вызвать make clean, чтобы удалить все старые объектные файлы. При последующей сборке все файлы кода будут собраны заново, что, вероятно, решит проблему. Например, когда мы в институте писали драйвер для двигателя, в какой-то момент у нас начала менять яркость экрана от ПЛК, и мы минут 20 пытались понять, что идёт не так, пока не решили пересобрать весь проект заново.

CMake

Главная проблема Makefile заключается в том, что он под Linux и его инструкции невозможно перенести без проблем. Для Windows придётся искать замену и переделывать команды (вместо rm -rf *.o был бы DEL /F *.o). Кросс-платформенных решений довольно много, на самом деле, но популярность диктует выбор, а значит придётся брать CMake. Для Windows есть установщик, для MSYS2 и Linux есть пакеты в пакетных менеджерах почти всегда. Проблема может возникнуть только с версией. Работа с CMake будет в первой программе.

Первая программа (Hello World без std::cout)

Настало время что-то всё-таки написать. Сделаем простую программу, которая будет выводить «Hello, World!» в консоль. Накидаем файликов в папку проекта:

📁 first-project
 ├📁 src
 │ └📄 main.cpp
 └📜 CMakeLists.txt
Заранее озадачимся тем, чтобы код приложения был отдельно от файлов сборки. 

«/first-project/src/main.cpp»
#include <print>

int main() {
  std::println("Hello, World!");
  return 0;
}

На самом деле здесь всё просто. В первой строке, через решётку (#) используется директива препроцессора, в данном случае «include». Это самая распространённая и необходимая директива в C/C++. Всё, что она делает — вставляет содержимое указанного файла на место её вызова, т.е. в процессе сборки данная конструкция развернётся и превратится в содержание файла «.../print», где-то в стандартных библиотеках C++. Например, на моём домашнем ПК с Visual Studio 2022 данный файл находится в C:\Program Files\Microsoft Visual Studio\2022\Community\VC\Tools\MSVC\14.43.34808\include\print. Те, кто учил C++, могли ожидать здесь страшную конструкцию std::cout << "Hello, World!" << std::endl;, но не стоит беспокоиться — данный ужас теперь в изгнании из уроков по плюсам на ближайшие лет пять точно. std::print может почти всё, что может std::cout, но пользоваться им куда удобнее. Содержание разберём после того, как успешно соберём программу, а для этого нужно написать инструкцию для CMake: 

«/first-project/CMakeLists.txt»
cmake_minimum_required(VERSION 4.0)

# Static linking of vcruntime
set(CMAKE_MSVC_RUNTIME_LIBRARY "MultiThreaded$<$<CONFIG:Debug>:Debug>")

# Put libs together with executables
set(CMAKE_RUNTIME_OUTPUT_DIRECTORY "${CMAKE_BINARY_DIR}/$<CONFIG>")
set(CMAKE_LIBRARY_OUTPUT_DIRECTORY "${CMAKE_BINARY_DIR}/$<CONFIG>")

set(CMAKE_CXX_STANDARD 23)
set(CMAKE_CXX_STANDARD_REQUIRED TRUE)

project(first-project LANGUAGES CXX
    VERSION 0.0.0.1
    DESCRIPTION "Our first project."
    HOMEPAGE_URL "https://github.com/VerMishelb/CppCmakeGit-learning-minefield/tree/master/tutorial-from-neocities/first-project"
)
add_compile_options(
    "$<$:/utf-8>"
    "$<$:/utf-8>"
)

# TARGETS
# first-program
add_executable(first-program src/main.cpp)
target_include_directories(first-program PUBLIC src)

??!

Давайте разберём, что здесь вообще происходит, двигаясь сверху вниз, обозначив основные единицы синтаксиса.

  • В snake_case обычно пишутся функции (add_compile_options(), set(), if()).
  • ПРОПИСНЫМИ пишутся переменные и свойства.
  • Комментарии начинаются с # решётки.
    Многострочные начинаются с #[=[ и заканчиваются на ]=]. Количество = может быть любым, включая 0, но открывающая и закрывающая последовательности должны совпадать по количеству знаков.
  • Переменные используются с помощью ${ТАКОГО} синтаксиса.
  • Всё, что выглядит $<ТАК> или $<ВОТ:ТАК> — выражение генератора CMake.
Сам CMake во многих местах на регистр букв не смотрит, поэтому ничто не мешает написать ADD_PROJECT(). Для читаемости лучше так не делать.

cmake_minimum_required(VERSION 4.0)
Данная строка должна быть в корневом CMakeLists.txt проекта, т.е. тот «лист» (будем так их называть), который вы находите первым, когда открываете папку с проектом. Она задаёт минимальную поддерживаемую версию CMake, на которой может быть собран этот проект. Часто проекты не используют самые новые функции CMake, из-за чего можно обойтись версией постарше, повысив совместимость.

set(CMAKE_MSVC_RUNTIME_LIBRARY "MultiThreaded$<$<CONFIG:Debug>:Debug>")

  • set(ПЕРЕМЕННАЯ ЗНАЧЕНИЕ) создаёт переменную, если её ещё нет и задаёт ей значение.
  • CMAKE_MSVC_RUNTIME_LIBRARY — одна из стандартных переменных CMake, которая задаёт параметры для среды исполнения MSVC, точнее для того, с чем вы, как пользователь, сталкивались под названием «vcredist». Библиотека среды исполнения может быть в одном из четырёх видов: MultiThreaded, MultiThreadedDll и ещё два таких же, но отладочных. Отличие значений с суффиксом Dll в том, что приложение будет использовать системные *.dll файлы среды исполнения вместо того, чтобы привязать их статически. Это уменьшает вес исполняемого файла, но без установки vcredist запустить его не получится.
  • MultiThreaded$<$<CONFIG:Debug>:Debug> — значение для переменной. Первая часть MultiThreaded простой текст, а $<$<CONFIG:Debug>:Debug> — «выражение генератора» CMake. $<CONFIG> — специальное выражение генератора, которое превращается в Debug, Release, RelWithDebInfo или MinSizeRel в зависимости от собираемой конфигурации (перечислены стандартные, но могут быть и свои).
    Суть в том, что $<CONFIG:Debug> становится 1 при сборке конфигурации Debug. Дальше полученное значение используется в другом выражении $<$<CONFIG:Debug>:Debug>, которое принимает (текстовое) значение Debug, если конструкция в левой части выражения равна 1.
    Получается, что в зависимости от конфигурации задаётся значение MultiThreaded или MultiThreadedDebug.
Целиком вся эта конструкция статически привязывает библиотеку среды исполнения MSVC к исполняемым файлам, чтобы изделие запускалось без установки дополнительных компонентов системы. Если вам это не кажется серьёзной проблемой, задумайтесь над тем, что на моей текущей системе (установлена весной 2023 года) стоит уже 15 разных версий Microsoft Visual C++ Redistributable, от 2005 года до 2022. Весят они, конечно, мало, но каждая ставилась отдельно.

set(CMAKE_RUNTIME_OUTPUT_DIRECTORY "${CMAKE_BINARY_DIR}/$<CONFIG>")
set(CMAKE_LIBRARY_OUTPUT_DIRECTORY "${CMAKE_BINARY_DIR}/$<CONFIG>")
CMAKE_RUNTIME/LIBRARY_OUTPUT_DIRECTORY — куда класть результаты сборок приложений (.exe, .elf) и библиотек (.dll, .so).
Задаются вручную, потому что при использовании, скажем, SDL, без этих переменных проект библиотеки и проект приложения соберутся в разных директориях, а значит исполняемый файл не найдёт рядом с собой какую-нибудь dll-ку (если вы, конечно, не связали их статически, что не всегда возможно).
Ninja Multi-Config, MSVC и прочие системы сборки, которые используют конфигурации, будут создавать свои подпапки. Их можно остановить, обернув значение переменной в выражение генератора: $<1:"${CMAKE_BINARY_DIR}/$<CONFIG>"> (1 здесь как TRUE).

set(CMAKE_CXX_STANDARD 23)
set(CMAKE_CXX_STANDARD_REQUIRED TRUE)
CMAKE_CXX_STANDARD задаёт стандарт C++ для использования, а _REQUIRED просит без этого стандарта даже не пытаться собирать проект. Просит вроде как глобально, НО! Поскольку наш проект использует один и тот же стандарт языка для всех файлов с кодом, а в случае написании библиотеки этот лист перестанет быть верхним уровнем, следовательно не повлияет на включающий проект, ничего страшного здесь нет. Да и наш проект — не библиотека. 

project(first-project LANGUAGES CXX
    VERSION 0.0.0.1
    DESCRIPTION "Our first project."
    HOMEPAGE_URL "https://github.com/VerMishelb/CppCmakeGit-learning-minefield/tree/master/tutorial-from-neocities/first-project"
)
  • project() — команда для создания проекта. В каждом проекте должна быть одна. От него всё отсчитывается. Формально, это допускает существование вложенных проектов в пределах одного большого, но на практике я такого не видел.
  • LANGUAGES — после этого слова задаются языки программирования, через пробел. В нашем проекте только плюсы, поэтому здесь CXX.
  • VERSION — версия проекта. Примечательно, что версия есть только для проекта, но не для исполняемых файлов, добавляемых отдельно. Странная система.
  • DESCRIPTION — описание проекта.
  • HOMEPAGE_URL — домашняя страница проекта.
Формально, здесь можно обойтись и project(first-project), но мне хотелось показать, каким образом можно задавать свойства, указанные как опциональные аргументы в документации CMake. 

add_compile_options(
    "$<$<C_COMPILER_ID:MSVC>:/utf-8>"
    "$<$<CXX_COMPILER_ID:MSVC>:/utf-8>"
)
add_compile_options() устанавливает (точнее, добавляет к существующим) флаги для компилятора по умолчанию. Здесь используется уже знакомая нам конструкция с условием через выражение генератора, чтобы задать флаг /utf-8, если сборка производится компилятором MSVC, потому что по умолчанию он почему-то использует системную кодировку (1251).

add_executable(first-program src/main.cpp)
Добавляет в проект новую цель в виде исполняемого файла, т.е. для Windows это будет first-program.exe (расширение он сам при сборке додумает). После указания названия цели перечисляются файлы кода для сборки.

target_include_directories(first-program PUBLIC src)
Команда добавляет к цели директории, в которых можно искать заголовочные файлы (.h, .hpp).
first-program — цель.
PUBLIC контролирует видимость того, что мы делаем командой. В данном случае контролирует то, будут ли другие проекты, использующие наш как библиотеку, видеть папку src как папку с заголовочными файлами. Бывает PRIVATE (нужно включить директорию для сборки цели, но другим целям она не нужна), INTERFACE (не нужна для сборки, но нужна другим проектам (например, интерфейс взаимодействия с библиотекой, вынесенный в отдельные заголовочные файлы)) и PUBLIC (нужно дать доступ всем).
src здесь директория, в которой мы ищем файлы. Формально это папка с кодом (.cpp), но когда заголовочный файл не нужен другим проектам, мне удобнее оставить его в src.

Перед тем, как идти собирать проект, нужно открыть настройки VSC, найти там пункт cmake.outputLogEncoding и задать ему значение utf8, чтобы в консоли можно было прочитать ошибки при сборке.

Но это в том случае, если вы собираетесь использовать VSCode. А без него придётся заниматься этим вручную, либо написать несколько скриптов: 
configure.bat
cmake -S . -B build -D CMAKE_BUILD_TYPE=Release
-S указывает директорию исходников (где лежит главный лист с project). Раз скрипт лежит в той папке, которая нам нужна, просто указываем точкой текущую директорию.
-B указывает директорию, где будут расположены файлы системы сборки (они — результат конфигурирования проекта). Если вы используете Git, эту папку стоит добавить в .gitignore.
-D задаёт значение переменной CMake. Если ещё точнее, то запись в кэше. Пробел между -D и названием переменной опционален.
Помимо этих есть ещё -G, -A, -T и прочие, про которые лучше читать в документации. 
build.bat
cmake --build build --config Release -j
У CMake есть режим сборки, который освобождает вас от необходимости разбираться с системой сборки, которая используется.
После --build указывается папка с файлами системы сборки (-B из прошлой команды).
--config задаёт тип сборки, если система поддерживает несколько конфигураций (MSVC, Ninja Multi-Config). Если не поддерживает, тип задаётся на этапе конфигурирования проекта с помощью CMAKE_BUILD_TYPE (см. выше).
-j без параметров просит CMake всё-таки использовать 24 потока моего процессора вместо одного. Если после -j написать число, будет использовано заданное кол-во потоков. 
clean.bat
cmake --build build --target clean
Сборка цели clean очищает проект от артефактов сборки. Я уже писал вышел про экран ПЛК. С другой стороны, ничто не мешает удалить всю папку build и собрать её заново.

Всё. Выполнение конфигурирования и сборки создаёт исполняемый файл first-project\build\Release\first-program.exe, при выполнении которого в консоль выводится сообщение. Если сборка производилась из VSCode, скорее всего, вместо Release будет Debug, если только вы сами не поменяли тип сборки.
